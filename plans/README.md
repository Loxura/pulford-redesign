## Audited but unplanned findings (ask for a plan if wanted)

- **A11Y batch (S, HIGH confidence)**: (a) shop.html booking-bag dialog has
  `aria-modal="true"` but no focus trap and no `inert` on the background
  (shop.html:301 + inline script `open()`/`close()` at ~461–467); (b) the
  theme toggle's `aria-label` is stale on first load when dark theme is
  persisted (static `aria-label="Switch to dusk"` in every header; app.js
  only updates it on click, app.js:13); (c) under `prefers-reduced-motion`
  the hero WebGL shader renders one frame and never re-renders after a theme
  toggle, leaving a daylight-coloured canvas on a dusk page (app.js:122,127
  — the rAF loop is gated on `!reduce`).
- **DEBT: header/footer/SVG duplication (M, MED risk)**: identical header
  (~26 lines), footer (~45 lines incl. the village SVG), and the door-icon
  SVG (~17 inline copies) are duplicated across all 7 site pages; the CTA
  doors SVG appears 3× (index/news/get-involved) and the emblem SVG 2×
  (about/get-involved). Every global edit touches 7 files. Deferred
  deliberately: a runtime-include refactor restructures the pitch artifact
  close to the pitch date, and `fetch()`-based partials break on `file://`.
  Decide after the pitch — or fold into the planned Next.js rebuild, where
  componentization comes free.
- **DEBT: shop price duplication (S)**: prices exist in shop.html card
  markup (lines 90–195) AND in the inline `PRICES` map (shop.html:390–398).
  Editing one without the other silently desyncs the displayed price from
  the bag total. Cheap fix: render `.price-row` from `PRICES` on load.

## Findings considered and rejected

- WebGL context-loss handling in the hero shader: not worth it for a
  prototype — the CSS radial-gradient fallback in `.hero-door` already
  covers a dead canvas.
- `window.__pfPauseShader` (app.js:116) is dead code from the shelved
  door-loader concept: harmless; remove opportunistically if app.js is
  edited anyway, not worth its own change.
- TipTap pinned via esm.sh import map "tech debt": by design for a concept
  build; superseded by plan 003 vendoring it.
- News cards being dead ends (`.go` spans aren't links, news.html): by
  design for a mockup — re-cast as direction finding D1 below.
- Missing favicon/Open Graph tags: cosmetic for a local pitch demo; becomes
  relevant only if the prototype is hosted publicly (see D3).
- `git add` of `mockup/_shots/` (~14 MB of PNGs) bloating the repo: accepted
  — it's the only design record; revisit only if a remote with size limits
  enters the picture.

## Direction findings (maintainer's call, not ranked against bugs)

- **D1 — Article detail page closing the CMS loop (M)**: the CMS demo
  (cms.html) edits the exact story teased on news.html ("the same story the
  CMS demo is editing", news.html:55), but no public article page exists, so
  the pitch beat "type something, hit Publish" ends in dead air. One
  `article.html` template rendering that story would let the pitch show
  publish → live. Trade-off: one more page to keep in sync with the
  duplicated header/footer.
- **D2 — Mock donate flow (S–M)**: every page funnels to
  `get-involved.html#donate`, whose buttons currently link to contact.html;
  the copy promises CanadaHelps/Donorbox. A lightweight mock donate modal
  (amount picker → fake confirmation, clearly labelled demo) would let the
  pitch demonstrate the giving journey end-to-end. Trade-off: risks looking
  like a real payment promise; needs the same "demo" framing as the booking
  bag.
- **D3 — Host the prototype (S)**: putting `mockup/` on Netlify/GitHub Pages
  with a QR code lets board members browse on their phones after the pitch.
  Depends on plans 002 (touch CTAs) and 004 (payload) landing first;
  pairs with adding favicon/OG tags. Requires the user's call on making it
  public.
